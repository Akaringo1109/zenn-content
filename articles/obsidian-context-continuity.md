## はじめに

3歳と1歳を育てながら、外資コンサルでAI導入をリードしているあかりんごです。

少し前まで、「先週の会議で何が決まったんだっけ」を思い出せずに、議事録を漁る時間を毎日のように溶かしていました。

私の一日は、だいたい決まっています。
早朝4時台に起きて走り、上の子を幼稚園へ、下の子を保育園へ送り、そこからフルスロットルで仕事に入ります。
複数のプロジェクトを横断していて、頭のなかは常に何かで埋まっています。

昔は「全部覚えておこう」としていました。
要件は Excel で必死に整理し、決定事項は記憶と個人メモに頼る。
でも、抱える案件が増えるほど、この方法は破綻しました。

あるとき、気づきました。
これは記憶力の問題ではなく、 **コンテキストが途切れる** 問題だったんです。

この記事は、その解き方を「抽象論」で終わらせずに、私が実際に使っている **フォルダ構成・テンプレート・プロンプト・運用の中身** まで公開します。
案件名や顧客名はマスキングしていますが、構造や仕組みはすべて実物です。

内容を2本に分けました。
この **ためる編** では、コンテキストを漏れなく「外部脳」に貯める工夫を。
貯めた文脈を AI で引き出し、振り返りに活かす **つかう編** は近日公開予定です。

:::message
この記事に載せる Templater・デイリーノートのテンプレートや議事録プロンプトは、私が普段使っている実物です。そのままコピーして自由に使ってください（無料）。
:::

前提はシンプルです。

- Obsidian を常時開いた「マスター」として運用（PARA のフォルダ構成）
- 会議の議事録・チャットのログ・案件の資材を、すべてそのなかに格納
- その上で Claude Code を動かし、AI が常に案件の文脈を持った状態で私と対話する

エンジニアではないけれど上流の仕事をしていて、AI を業務に食い込ませたい方に届けば嬉しいです。

## 考え方：なぜ、すべてを外部脳に集めるのか

まずは、細かい運用の前にある「考え方」から。
ここが腑に落ちれば、あとの具体は全部その応用です。

ひと言でまとめると、私がやっていることは **コンテキストエンジニアリング** です。

よく聞く「プロンプトエンジニアリング」は、AIへの一回の問いかけを磨く技術です。コンテキストエンジニアリングはその手前にある問いで、「AIが正しく判断できる文脈を、どう設計・維持するか」を扱います。どれだけ精巧なプロンプトを書いても、渡す文脈が貧しければ、答えは貧しいままです。

私のObsidianの構造・テンプレート・自動化のすべては、 **コンテキストが途切れないように設計し、それを実践し続けるための仕組み** です。
ためる編では、その設計の中身を公開します。

![コンテキストエンジニアリングを支える3つの考え方](/images/obsidian-context-continuity/context-engineering-3pillars.png)

### すべてのコンテキストをAIが読める状態にする（会議・チャット・対面のログ化）

貫いている一本の原則があります。

To Do も決定事項も、重要なコンテキストも、ほぼ間違いなく会議・チャット・対面のどこかから生まれます。
そして、その多くは流れて消えます。
だったら、消える前に、発生源をそのまま貯めればいい。
要約ではなく、一次情報を貯める。ここだけは徹底しています。

なぜ要約ではダメなのか。
私は AI を「うちのルールをまだ知らない、優秀な新人」だと思っています。
どれだけ優秀でも、この案件で半年前に何が揉めたかは知りません。
要約だけ渡しても、深い判断はできません。
だから、発言のニュアンスまで含んだ生の文脈を、丸ごと渡せる状態にしておきます。

### Obsidian を「マスター」にして、すべてを一か所に集める

集めるのは、議事録・チャット・資材だけではありません。
**外部のソースリポジトリも、この Vault のなかに複数インポートしています。**

Obsidian をひとつの「マスター」と位置づけて、そこに案件のドキュメントも、実際のソースコードも同居させる。
すると何が起きるか。
チャットで「これ、どういう仕様でしたっけ？」と聞かれることがあります。
議事録に「こう改修してほしい」と書かれていることもあります。
そんなとき、Obsidian 上で動くエージェントは、同じ Vault にあるソースを開いて、調査も改修もその場でやってくれます。

会議の一言から、ソースコードの中身まで。
これが地続きになるのが、マスター Vault にする狙いです。

### 共有したいルールは、共通コンテキストファイルでセッション横断させる

もうひとつ、基本の型があります。
**AI と共有したいことは、会話で毎回説明せず、共通コンテキストファイルとして書いておく** 、というものです。

口頭の指示はその場で消えます。
でも、ファイルに書いたルールは毎回読まれます。
判断基準も、繰り返す作業の手順も、共通ファイルに落としておけば、AI は私と同じ前提で動けます。
このあと出てくるタスクの緊急度ルールは、その一例です。具体的には CLAUDE.md や Markdown 形式のスラッシュコマンドがこの役割を担っています。

## 漏れなく「ためる」仕組み

ここからは、実際にどう貯めているかの具体です。

### 私の作業環境：Obsidian の実画面

話が具体に入る前に、今日の作業画面をそのまま見せます。

![左からグラフビュー・フォルダツリー・デイリーノート・カレンダーが並ぶ実際の作業画面（案件名はマスキング）](/images/obsidian-context-continuity/Obsidian-env.excalidraw.png)

左端にグラフビューとフォルダツリー（タグ一覧含む）、中央にデイリーノートと Thino（デイリーメモ）、右端にカレンダーとテンプレート呼び出し。
常時4〜5ペインを並べて、視点を切り替えながら動いています。
見えているものの意味は、この後の具体で順に出てきます。

### 外部脳が育つと、グラフビューに現れる

ここで一度、「続けた先」を見てほしいと思います。

![私のグラフビュー。クラスタが案件・関心領域に対応する](/images/obsidian-context-continuity/graph-view.png)

一つひとつの点がノートで、線がリンクです。
かたまり（クラスタ）ごとに案件や関心領域が分かれていて、どのテーマにどれだけ情報が集まっているかが、ひと目で見えます。
これはそのまま、私の頭のなかの外部コピー、つまり **脳の外部化** です。

このグラフは、放っておくと孤立したノートが増えます。
なので `/smart-links`（Smart Connections の意味検索で、内容の近い未リンクのノートを見つけて自動で繋ぐスラッシュコマンド）で、ときどき繋ぎ直して育てています。

どうやってこの外部脳を育てるか。以降の具体で順に説明します。

### フォルダはPARAで、時間軸は自動で仕分ける

まず、器になるフォルダ構成です。
PARA（Projects / Areas / Resources / Archive）をベースに、番号プレフィックスで並び順を固定しています。

```
<Vault>/
├── 00 - DailyNotes/   # 日々のログ・タスクの一時置き場
├── 05 - Inbox/        # とりあえずの受け皿（常に空を目指す）
├── 10 - Projects/     # 進行中の案件（ここが本体）
├── 20 - Areas/        # 継続的な責任範囲・関心分野
├── 30 - Resources/    # 参照資料・Webクリップ
├── 40 - Archive/      # 完了・非アクティブ
├── 50 - Output/       # 生成物（記事・資料）
├── 90 - Attachments/  # 画像・PDF など
└── 95 - Config/       # テンプレート・設定
```

いちばんアクセスするのが `10 - Projects/` で、ここに案件のすべてが集まります。

```
10 - Projects/
├── _MeetingNote/   # 全案件の議事録を1か所に集約
├── _Chat/          # 全案件のチャットログを1か所に集約
└── {案件名}/
    ├── {案件名}.md    # ハブノート（配下を wikilink で束ねる）
    ├── {案件名}.base  # Obsidian Base（動的な一覧ビュー）
    ├── 2026-01-06_{トピック}.md   # 「日付_トピック」で残すメモ・議事録
    ├── 設計/
    └── 10_src/{repo}/  # 取り込んだソースを議事録の隣に同居させる
```

時間軸のノートは、置き場所を自分で考えません。
デイリーノートを新規作成すると、Templater のスクリプトが自動でこのパスへ動かしてくれます。

```
00 - DailyNotes/2026/07/3/2026-07-14.md
                  │   │  └ その月の第何週
                  │   └── 月
                  └────── 年
```

「今日のノートをどこに置くか」を毎日考えないための、ちょっとした仕込みです。
そしてこの週フォルダが、週次レポート生成の集計単位にもなります。

### 会議：議事録と発言記録を週次で積む

ここが、一次情報の入り口です。

うちの会議はほぼ Teams で行っています。
議事録は、手で書きません。
Microsoft 365 Copilot の自動議事録機能に、 **定型プロンプト** を食わせて生成しています。
毎回使っているプロンプトの本体が、これです。

```
Think Deeper で実行してください。

会議内容から、日本語の議事録を作成してください。雑談・チャットの内容は反映しません。

議事録は Markdown 形式で、以下の流れで構成されます。

まず冒頭に、会議名・開催日時（JST）・参加者（アルファベット表記）を記載します。

続く最初のセクションは「0. 会議全体のサマリー」で、その日の議論を俯瞰し、何が話され、どこに重点があったかを簡潔に整理します。

次のセクション「1. 決定事項」では、会議の中で合意・確定したことを箇条書きで列挙します。

その次の「2. ToDo」では、会議から発生したアクションを箇条書きで記載します。各 ToDo は「内容／担当／期限」を1行にまとめ、不明な部分は「不明」と書きます。数を増やすことよりも、抜け漏れや重複のない質の高いリストにすることを優先します。

続く「3. 議題ごとのセクション」では、議論された議題を順に番号付きで並べ、各議題で「主な論点」と「議論の詳細」を中見出しとして展開します。発言を引用する際は文末に（発言者）の形で発言者名を添え、その発言から派生した ToDo がある場合は、発言末尾に該当 ToDo の番号を紐付けて示します。

最後の「4. 非参加者向けの一言での理解」では、この会議に参加していない人にも本質が伝わるよう、「この会議を一言で言うと〜」の形で要約し、なぜそれが重要なのかを短く補足します。

議事以外（思考・提案・補足コメント）は一切出力しません。
```

ポイントは、 **決定事項** と **ToDo（内容／担当／期限）** が必ず同じ型で出てくることです。
毎回この構造で吐かれるので、あとから「誰の・いつまでの」タスクかを機械的に拾えます。

生成された議事録は、Obsidian 側の会議ノートテンプレートに流し込みます。
このテンプレートも、デイリーノートと同じく、作った瞬間に週フォルダへ自動で仕分けられます。

```
10 - Projects/_MeetingNote/2026/07/3/2026-07-15_{会議名}.md
```

テンプレートの中身はこうです。

```markdown
---
tags: [meeting, project]
---
# 議事録：（会議名）

## 0. 前回からのTodo
| No. | Title | Description | PIC | Status | Due |
| --- | ----- | ----------- | --- | ------ | --- |

## 1. 会議情報
- 日時 / 場所 / 出席者

## 2. アジェンダ
## 3. 議論内容
## 4. 決定事項
## 5. アクションアイテム
| No. | Title | Description | PIC | Status | Due |
| --- | ----- | ----------- | --- | ------ | --- |

## 6. 次回会議・その他メモ
## 7. 関連ノート
（末尾の dataviewjs が、同じタグ・被リンクのノートを自動で列挙）
```

「0. 前回からのTodo」と「5. アクションアイテム」を表で持っているので、前回の宿題と今回の宿題が、担当・期限つきで毎回そろいます。

:::details 週フォルダへ自動仕分けする Templater スクリプト

```js
// 会議ノート末尾の Templater。作成日から週を求め、_MeetingNote/年/月/週/ へ移動する
const now = new Date();
const year = now.getFullYear();
const month = String(now.getMonth() + 1).padStart(2, "0");

const weekStart = new Date(now);
weekStart.setDate(now.getDate() - now.getDay());   // その週の日曜
const firstSunday = new Date(year, now.getMonth(), 1);
firstSunday.setDate(1 - firstSunday.getDay());      // 月初の日曜
const week = Math.floor((weekStart - firstSunday) / (7 * 864e5)) + 1;

const dir = `10 - Projects/_MeetingNote/${year}/${month}/${week}`;
for (const p of [`10 - Projects/_MeetingNote/${year}`, `10 - Projects/_MeetingNote/${year}/${month}`, dir]) {
  if (!app.vault.getAbstractFileByPath(p)) await app.vault.createFolder(p);
}
const prefix = `${year}-${month}-${String(now.getDate()).padStart(2, "0")}`;
await tp.file.move(`${dir}/${prefix}_${tp.file.title}`);
```
:::

そして議事録には、3つの軸でタグを付けています。
（案件名・クライアント名はマスキングしています）

```yaml
tags:
  - project/report-automation   # 案件ごと（複数可）
  - client/acme-corp            # クライアント
  - meeting-type/sync           # 会議の性質
```

会議の性質（`meeting-type`）は、`sync`（定例・朝会）／`discussion`／`review`／`kickoff`／`1on1`／`qa`／`training` といった値で分類しています。
このタグ付けは、ファイル名だけを見てルールベースで機械分類し、判定できなかったものだけ AI に回す **ハイブリッド** です。
本文まで読ませると、他案件の名前がチラ見えしてタグが汚染されるので、あえて本文は見ません。

議事録を週単位のフォルダに積み重ねることで、「先週の会議をまとめて」「先月この案件でどんな議論があったか」のような問いに、AI が漏れなく答えられるようになります。
週フォルダが集計単位として機能するので、週次レポートの生成も `/progress-report` のひと呼びで完結します。
いま議事録は500本を超えました。

**会議でもう一つ：自分の発言も記録する**

貯めているのは、決定事項や ToDo だけではありません。
**自分自身が、その会議でどう振る舞ったか** も、同じ Copilot で記録しています。

さきほどの Copilot には、議事録用とは別に、もう1つプロンプトを渡しています。
「本会議での立ち位置」から始まる、自分の発言だけをナラティブに書き起こすためのプロンプトです。
個人名は伏せますが、こんな構造で自分の発言記録を残しています。

![1つの会議から、自分の発言を4つの観点で記録し、振り返りに使う](/images/obsidian-context-continuity/self-speech-record.png)

```
## 発言記録
### 本会議での立ち位置
ファシリテーターか、提案する側か、判断を仰がれる側か。どんな姿勢で臨んだか。

### 議題別 発言ナラティブ
何を問題視して発言し、議論をどう動かしたか。認識がどう変わったか
（当初〜と捉えていたが、〜を経て〜へ切り替えた、の形で変遷を残す）。

### この会議での判断・コミット
何を承認／合意／却下／保留したか。その判断の"質"（積極的な合意か、消極的な追認か）まで。

### 通底する人物シグナル
複数の会議を横断して見えてくる、自分の思考の癖・譲れない一線。
```

決定事項の記録が「プロジェクトの外部化」なら、これは **自分自身の外部化** です。
何を考え、どう判断したかが、会議のたびに溜まっていく。
この記録を実際にどう振り返りに使うかは、つかう編で書きます。

### チャット：週次ログを自動で積む

チャットは、自作のブラウザ拡張で書き出しています。
Teams の画面を Markdown に変換し、 **チャネルごと・週ごと** のファイルに割って保存します。

```
10 - Projects/_Chat/{チャネル名}/2026-07-W03.md
```

週単位でファイルを区切っているのは、AI が扱いやすい粒度に揃えるためです。
「この週のこのチャネルで何が話されたか」をひとつのファイルで渡せると、週サマリーや案件をまたいだ横断集計をしやすくなります。
会議ログと同じく、週フォルダが集計単位として機能します。

チャットは100以上のチャネル・週次ログにして600ファイル、発言数にすると5,000件をとうに超えました。
消えていくはずだった会話が、案件ごとに構造化されて積み上がっています。

### デイリーノートは「入力バッファ」として割り切る

デイリーノートには、凝ったことを書きません。
断片を放り込む **入力バッファ** として割り切っています。

![サンプルのデイリーノート。フォーマットとタスク構造を再現した架空の内容](/images/obsidian-context-continuity/daily-note-screenshot.png)

この画像は、フォーマットをそのまま再現したサンプルです。プロジェクト名・タスク内容はすべて架空のものです。

---

# [[<% tp.date.now("YYYY-MM-DD", -1, tp.file.title, "YYYY-MM-DD") %>|◀]] <% tp.date.now("YYYY年MM月DD日") %> <% 曜日 %> [[<% tp.date.now("YYYY-MM-DD", 1, tp.file.title, "YYYY-MM-DD") %>|▶]]

---
## 📝 今日のTASK
- [ ]

---
## 分報

---
## タスク消化実績
### ✅ 完了タスク
### 🚨 未完了タスク

---
## 🔄 YWTふりかえり
### 🚀 Y：やったこと＋工夫したこと
### 🧠 W：得た気づき・解釈

## 💡 気づき・学び（Insights）

# 🔗 関連ノート
（末尾の dataviewjs が、同じタグ・被リンクのノートを自動で列挙）

冒頭の日付行に Templater を仕込んでいて、 **前日・翌日への移動リンク（`[[◀]]` `[[▶]]`）と曜日が自動** で入ります。
これで日付が一本の線で繋がり、時系列を遡れます。
さらにノート末尾にもう一つスクリプトを置いていて、作成した瞬間に `年/月/週/` のフォルダへ自動で仕分けます。

:::details 関連ノート用 dataviewjs と、自動仕分け Templater スクリプトの全文

```js
// # 🔗 関連ノート の下に置いている dataviewjs
var maxLoop = Math.min(dv.current().file.tags.length, 5);
for (let i = 0; i < maxLoop; i++) {
  dv.span(dv.current().file.tags[i]);
  dv.list(dv.pages(dv.current().file.tags[i]).sort(f => f.file.mtime.ts, "desc").limit(15).file.link);
}
for (let outgo of dv.pages('outgoing([[' + dv.current().file.name + ']])')) {
  dv.header(4, outgo.file.name);
  dv.list(outgo.file.inlinks.sort());
}
```

```js
// ノート末尾の Templater。作成時に 00 - DailyNotes/YYYY/MM/週番号/ へ自動移動する
const fileName = tp.file.title; // 例: "2026-07-14"
const [year, month, day] = fileName.split("-");
const date = new Date(`${year}-${month}-${day}`);

// その週の日曜日
const weekStart = new Date(date);
weekStart.setDate(date.getDate() - date.getDay());
// 月初の日曜日
const firstDay = new Date(year, month - 1, 1);
const firstSunday = new Date(firstDay);
firstSunday.setDate(firstDay.getDate() - firstDay.getDay());
// 月の第何週か
const weekNumber = Math.floor((weekStart - firstSunday) / (7 * 864e5)) + 1;

const weekPath = `00 - DailyNotes/${year}/${month}/${weekNumber}`;
for (const p of [`00 - DailyNotes/${year}`, `00 - DailyNotes/${year}/${month}`, weekPath]) {
  if (!app.vault.getAbstractFileByPath(p)) await app.vault.createFolder(p);
}
await tp.file.move(`${weekPath}/${fileName}`);
```
:::

タスクには、緊急度を装飾で持たせています。
装飾は見た目のためではなく、機械が読める状態にするためです。

- 当日中に対応：`==テキスト==`（ハイライト）
- 3日以内：`**テキスト**`（太字）
- それ以外：装飾なし

この装飾ルールは、CLAUDE.md にも同じ内容を書いて AI と共有しています。
だから AI も `==` を「今日やる」、`**` を「3日以内」と、私とまったく同じ意味で読みます。
さきほど触れた「共有したいことは Markdown に書く」の、いちばん小さな一例です。

## うまくいっていないこと：ためる編の宿題

ここまで書くと順調そうですが、貯める側にも綻びがあります。
自分で決めた原則から、自分で外れているものばかりです。

![「こうすべき」という原則と、実際できていない現実のズレ](/images/obsidian-context-continuity/principle-vs-reality.png)

### リアルタイム・対面の会議は、まだ取りこぼしている

Teams のオンライン会議は、Copilot で議事録が取れます。
でも、対面の打ち合わせや、廊下での立ち話、電話は、コンテキストとして貯められていません。
いちばん熱量の高い決定が、実は対面で起きていたりするのに、です。

Plaud のような録音・文字起こしデバイスで、この「その場で消える会話」も外部脳に流し込みたい。
でも、まだそこには手が回っていません。ここは宿題です。

### Inbox が、ぜんぜん空にならない

私のルールでは、Inbox は「常に空を目指す」場所です。
一時的に放り込んだら、案件フォルダへ割り振って、済んだら Archive へ動かす。そのはずでした。
現実は、30本以上が割り振られないまま滞留し、Archive には十数本しか入っていません。
「あとで整理する」の「あとで」が、いちばん来ません。

### フォルダ階層が、深くなりすぎた

本来、ノート術の思想（Zettelkasten など）では、フォルダは浅くフラットに保ち、タグとリンクで繋ぐのが原則です。
でも私の案件フォルダは、案件 > サブ案件 > カテゴリ > … と、深いところで5〜6階層まで潜っています。
「リンクで繋ぐ」思想と、「フォルダで分ける」現実が、静かにぶつかっています。
